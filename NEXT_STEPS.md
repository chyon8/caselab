## 내일 할 것 (순서대로)

### 1. projects 백필 완주

`LIMIT`을 **200~500**으로 올리고 루프를 돌린다. 100건에서 정상 동작 확인됨.

> ⚠️ **500 이상은 권하지 않는다.** `description`이 최대 5,000자라, 긴 공고가 몰린 배치는
> 요청 본문이 Vercel 한도(4.5MB)를 넘겨 터진다. 그러면 커서가 안 움직여 **같은 배치를
> 무한 재시도**하며 백필이 그 자리에 멈춘다. 총 소요 시간은 200이나 500이나 몇 분으로 비슷하다.

백필 후 바로 확인할 것:
- `manager_semin`이 화면에 계정명 그대로 보이는가? → 보이면 김세민의 실제 계정명이 다른 것 ([managers.ts](./src/lib/managers.ts) 수정)
- `project_type='term_based'`인데 `term_type != 'month'`인 행이 있는가?
  → 있으면 기간제 판별 기준을 `project_type`으로 바꿔야 한다 (현재는 `term_type` 기준)

### 2. Q&A(댓글) 워크플로

SQL 준비됨 → [`n8n/qna_incremental.sql`](./n8n/qna_incremental.sql)

projects와 **똑같은 4노드 패턴**. 다른 것은 두 개뿐:
- ③ 조회 SQL이 `qna_incremental.sql`
- ④ POST 대상이 `/api/sync/timeline` (엔드포인트는 이미 구현 완료)

> ⚠️ **projects 백필이 끝난 뒤에 돌린다.** 프로젝트가 CaseLab에 없으면 그 댓글은
> `skipped`로 반환되고 커서가 전진하지 않는다 (다음 주기 자동 재시도되므로 데이터 유실은 없다).

### 3. 특약(subcontracts) 미러링 ← **과업 범위 추적의 유일한 원천**

`initial_budget`이 "고객이 처음 적은 값"이라 무의미하다고 확인되면서,
**계약 이후의 과업 팽창을 알 수 있는 방법이 특약밖에 남지 않았다.**

`agreement_price`는 총액이라 "5,300만원 원계약 + 채팅 특약 500만원"인지
"5,800만원 한 방"인지 구분이 안 된다. 특약 행(`sub_contract_subcontract`)을 그대로
미러링해야 **"이런 프로젝트는 채팅에서 scope가 팽창한다"**는 인사이트의 1차 데이터가 생긴다.

필요한 것: 테이블 1개(`subcontracts`) + n8n 워크플로 1개. 백필하면 과거 1년치가 한 번에 들어온다.
(`work_scope`는 자유 텍스트라, "무엇이 추가됐는지" 구조화는 AI 단계에서 한다)

### 4. 녹취(calls) — **차단됨. 결정 먼저.**

수신 엔드포인트(`/api/sync/calls`)는 이미 구현돼 있다. 막힌 건 정책이다:

- **§7-5: 통화 요약을 외부 클라우드(Neon)에 저장해도 되는가?** ← 개발팀 확인 필요
- 전화번호·녹취 원문은 애초에 전송하지 않는 설계다 (n8n 내부에서만 사용 후 폐기)
- 다만 **요약문에 고객 이름·대화 내용이 남는다.** 연락처는 스크럽되지만 이름은 안 지운다(확정).
  PII 기준상 이게 허용되는지 판단이 필요하다.
- §7-4: n8n → 통화 API(`192.168.10.217:8000`) 접근 가능 여부도 함께 확인

## 알려진 부채

**🟢 완료 리뷰가 DB에 저장되지 않는다 — 고치지 않기로 결정 (2026-07-13).**
저장 버튼을 눌러도 React 상태만 바뀌고 `PostgresDataSource.saveReview()`가 호출되지 않아
새로고침하면 사라진다. DB 테이블(`reviews`)과 어댑터 메서드는 이미 있고 **쓰기 API 라우트만 없다.**
지금은 우선순위가 아니라고 판단해 그대로 둔다. 필요해지면 API 라우트 하나만 붙이면 된다.

**🟡 공고문이 원문 한 덩어리로 보인다.**
줄바꿈은 보존되도록 고쳤지만(`white-space: pre-wrap`), 항목별 구분(배경/필요요소/산출물 등)은
AI 구조화가 있어야 가능하다 → 대기 결정 #2에 묶여 있다.
그전까지 상세 페이지의 공고문 섹션 제목들은 빈 칸으로 남는다.

**🟡 알림(notifications)은 항상 빈 배열.**
본진에 대응하는 원천 테이블이 없다. 별도 설계가 필요하다.

**🟡 모바일 반응형 미구현 (추후 작업).**
코드 전체에 **미디어쿼리가 하나도 없다.** 유동 레이아웃(flex/grid)뿐이라 좁은 화면에서 깨진다.
손봐야 할 곳:
- 목록 테이블 — 6컬럼 고정 그리드 (`ProjectList.module.css`)
- 칸반 — 5개 컬럼 가로 배치
- 사이드바 (`AppShell`) — 접힘 상태는 있으나 모바일 전용 동작 없음
- 상세 페이지 — 공고문·타임라인 2단 구성
- 필터 바 — 검색창 + Select 3개가 한 줄

### 유사사례 "집계 뷰" (2026-07-13 요청)

유사사례를 나열만 하지 말고, **그 묶음의 통계와 공통 패턴**을 보여준다.
"이런 프로젝트들은 보통 이렇습니다"를 매니저에게 바로 제시하는 화면.

**중요: 두 층으로 나뉘고, 아래층은 AI 없이 지금 데이터로 계산된다.**

**층 1 — 통계 (AI 불필요, 데이터만 쌓이면 즉시 가능)**

| 지표 | 계산 방법 | 필요한 것 |
|---|---|---|
| 계약률 | 유사 사례 중 `stage >= 3` 비율 | ✅ 지금 있음 |
| 취소율 · **취소가 터진 단계 분포** | `status='완료(취소)'` 비율, `cancel_stage` 분포 | ✅ 지금 있음 |
| 모집 예산 → 실제 계약금액 델타 | `budget` vs `contract_amount` | ✅ 지금 있음 |
| 평균 기간 · 예산대 | `term_days`, `budget` 중앙값 | ✅ 지금 있음 |
| **계약 후 과업 팽창률** | 원계약 대비 특약 증액 합 | ⏳ 특약 미러링 필요 (내일 3번) |
| 평균 모집 기간 | `date_start_recruitment` → 계약일 | ✅ 지금 있음 |

→ **이 층은 AI 프롬프트 검토를 기다릴 필요가 없다.** 백필 + 특약만 끝나면 바로 만들 수 있고,
   "이 유형은 계약률 40%, 취소는 주로 모집 단계에서, 예산은 평균 27% 올라감" 같은 화면이 나온다.

**층 2 — 공통 패턴·리스크 (AI 필요, 프롬프트 검토 대기)**

- 자주 터지는 리스크 (녹취·노트에서 추출 → `ai_insights.risk_tags` 빈도 집계)
- 반복되는 문제 유형 (`issue_log` 클러스터링)
- **과업이 어느 기능에서 팽창하는가** — 특약 `work_scope` 텍스트를 LLM이 읽어야 안다
  (컬럼이 아니라 자유 텍스트라 정규식으로는 못 뽑는다)
- 개발사가 자주 묻는 질문 (`qna` 클러스터링)

→ 대기 결정 #2(프롬프트 검토)에 묶여 있다.
