**증상:** 백필 후 여러 프로젝트의 Q&A가 통째로 비어 있다 (예: project 156821, 그 외 다수). 프로젝트 자체는 정상.

**실측 (2026-07-15):**

| Neon `timeline_events` source='qna' | **16,717건** |

| 커서값 | `...\|97447` — 그런데 Neon에 다음 `97448`이 이미 있다 |

**fallback 날짜 = 무죄(기각).** project 156821은 본진 댓글 12개 중 **마지막 2개만** 있고 앞 10개는 커서 이전인데도 없다. 시점 경계 문제였다면 특정 시점 이전이 통째로 빠져야 하는데 "앞 10개 없고 뒤 2개만"은 설명 안 됨. 화면 문제도 아님(Neon 자체에 없음).

**1순위 용의자 — n8n ③→④ 배치 트렁케이션.** ③ 조회는 배치 전량을 리턴했는데 ④ 적재로 넘길 때 일부만 실렸고, 커서는 배치 max id까지 전진 → 안 실린 행은 skip=0인 채 **영구 누락.** CaseLab 라우트/[`cursor.ts`](./src/lib/sync/cursor.ts)는 정상(실측). 원인은 n8n ③→④ 매핑/페이징.

**추가 의심 — 커서 레이스.** 97447인데 Neon에 97448 존재 → 동시/수동 실행으로 커서 꼬였을 가능성.

**고치기:** ①본진 id 집합 − Neon id 집합으로 346건 특정 → ②n8n 실행로그서 ③ `data.length` vs ④ `rows.length` 비교(다르면 원인 확정) → ③커서 단일 실행 보장(동시/수동 금지, 실제 POST된 max id로만 전진) → ④누락분 재적재(독립 워크플로라 projects 재백필 불필요).

## 확정된 결정

### 유사사례 카드 컷 = 상대 마진 대신 바닥값만 (2026-07-20)
카드(사람이 보는 목록/상세패널)는 "1위 대비 마진" 컷을 쓰면 안 됨 — 1위가 이상치(거의 중복 사례)일 때 마진을 아무리 키워도 2위 이하가 통째로 잘림(문서마다 1위가 튀는 정도가 달라 고정 마진으로 못 막는 구조적 결함, 실측 재현: 아티스트 플랫폼 공고 8건→2건). `CARD_CUTOFF`(`src/data/postgres.ts`)를 바닥값(0.5)만 적용으로 변경 — LIMIT 8로 이미 개수 제한되고 유사도가 화면에 찍혀 사람이 거르므로 무관 섞임 비용이 낮다는 논리.
**통계·검수팁 풀은 기존 상대컷(1위 대비 0.06) 유지** — 같은 결함이 있어 poolSize가 1까지 줄어드는 사례를 확인했지만, 사용자가 "지금 결과로 충분"하다고 판단해 보류. **재개 조건:** 1위가 아니라 2위 기준으로 마진을 재는 방식 검토(이상치인 1위가 창을 못 끌어올리게).

### 통화(calls) 트랙 후순위 확정 (2026-07-16)
by-phone 매핑이 구조적으로 부정확 — 파트너·매니저 번호가 여러 프로젝트에 걸치고 한 통화에 여러 프로젝트가 섞임. confidence=high여도 오염 확인(예: project 153821에 무관 통화 다수 혼입). 지급·정산 이슈가 내용 다수인데 검수 매니저 개입 범위 밖. 껍데기 행(빈 summary/transcript) 35.7%(226/633)로 신호 밀도 낮음.
**조치:** UI 제거(`ProjectDetail.tsx`의 CallCard 삭제) — 데이터는 `calls` 테이블에 **보존만**, 다운스트림 사용 금지. 전체 백필 보류. 재개하려면 client 통화 × `meeting_meeting`에서 번호가 프로젝트 하나에만 걸리는 경우로 좁혀 매핑 신뢰도 확보부터.

### 미팅 녹취 vs 통화 = 별개 소스 (2026-07-15)
| | `calls` (by-phone) | `meetings` (/api/meetings/) |
|---|---|---|
| 매칭 | 전화번호 → project_id 추측 | project_id **직접** 매칭 |
| 특징 | confidence 필터 필요, 남 프로젝트 섞임 | 오매핑 없음 |
| 참여자 | 매니저↔클라 or 매니저↔파트너 양자 | 매니저·클라·파트너 **3자 미팅** 전문 |

미팅 파이프라인은 구현·실행 완료(`meetings` 테이블 적재, ProjectDetail 렌더). `meeting_meeting`에 `client_cell_phone_number`·`partner_cell_phone_number` 있음 — **SELECT 금지.**

### 계약 업무범위 텍스트 미러링 보류 (2026-07-15)
`work_scope`/`work_detail` 미러링 안 함. 대신 특약 숫자(`total_price`·`date_contracted`)와 `milestone_milestone` 예정일/실제일만. 이유: ①최고 신호(필요요소 8축)는 `devScope`에 이미 있음, ②원문은 계약 어드민 링크로 접근 가능, ③원문 70~80%가 계약 공통 보일러플레이트(프로토타입 렌더로 확인), ④회고적 데이터라 실시간 검수 판단에 안 쓰임. 실물은 7~8단 중첩 malformed HTML(순수 텍스트라는 원가정 틀림).
**재개 조건:** 추출 파이프라인 착수 시 첫 타깃은 **"제외 범위"**(공고엔 있었는데 계약서 협의로 빠진 것) — 신호 최고. 나머지 보일러플레이트는 여러 계약이 모여야 자동 제거 가능.
**계약 첨부파일은 파싱 안 함** — `project_projectfile`은 S3 FileField라 n8n서 signed URL 못 뽑고, PDF/HWP 파서는 비용 급증.

## 미착수 백로그

### 공고문 검색 랭킹 — 도메인 명사가 프로젝트 유형 신호를 덮어씀 (2026-07-19 진단)

**증상 (사용자 실사용 리포트 + 실측 재현):** 공고문 검색에 "반응형 홈페이지(PC/모바일), 여행상품 소개 페이지…"를 넣으면 1위가 "유럽 전문 여행사 홈페이지"가 아니라 여행 **플랫폼**류가 상위권을 채운다. "여행상품"→"상품" 한 단어만 바꿔서 재검색하면 결과가 여행 플랫폼 무더기 → 브랜드/기업 홈페이지 구축 무더기로 통째로 뒤바뀜(실측 확인). 즉 **"어떤 종류의 일인가"(유형)보다 "무엇에 관한 글인가"(도메인)가 벡터를 지배**한다.

**원인:** 코퍼스 임베딩 입력이 `제목 + 공고문 7000자`(`scripts/embed-projects.mjs`)인데, 도메인 명사(예: "여행")는 본문에 수십 번 반복돼 임베딩 평균을 끌고 가는 반면, 유형 신호("반응형 홈페이지 구축")는 제목 한 번뿐이라 구조적으로 밀린다.

**같이 진단·수정한 인접 문제(완료, `707bd71`)** — 이 항목과 원인이 다르므로 별도 처리함:
- L1 키워드 검색이 랭킹 없이 날짜순이라 "유지보수" 검색 시 제목 히트가 상위 15건 중 2건뿐이었음 → 필드 가중치 관련도 점수로 정렬 수정 (총 히트 건수는 그대로, recall 손실 없음).
- 유사사례 풀(카드·통계·검수팁)에 유사도 하한이 없어 무관한 사례가 항상 30건 채워지던 것 → 1위 대비 상대 컷(`sim >= max(top1-0.06, 0.5)`)으로 수정. **절대 임계값은 쓰면 안 됨** — 쿼리마다 유사도 스케일이 이동해서 한 쿼리의 30위가 다른 쿼리의 8위보다 높은 경우가 실측됨.
- 검수 팁(`mergeReviewTips`)이 자명한 일반론("요구사항을 명확히 정의")을 내던 것과, 지금 검수 중인 공고를 안 보고 무관한 리스크(웹사이트 공고에 앱 심사 리스크 등)를 섞던 것 → 프롬프트에 자명성 금지 규칙 + 공고문을 인자로 전달해 관련성 판단하도록 수정.
- Q&A 요약(`ai_insights.qna_summary`)에 기술적 제약·실현가능성 지적을 담을 필드가 없어 근거가 잘려나가던 것 → `technicalNotes` 필드 추가(코드는 배포됨, **기존 4,384건 재추출은 미실행** — `node scripts/extract-qna.mjs 4400`, ~$1.2).

**이 항목(유형 vs 도메인)은 미해결.** 검토한 선택지 4개:

| | 방법 | 비용/작업량 | 비고 |
|---|---|---|---|
| **A (추천)** | gpt-4o-mini로 유형 분류(신규 홈페이지/플랫폼·서비스/앱/유지보수·고도화/퍼블리싱·디자인 등) 컬럼 추가 후 `dev_scope`와 같은 소프트 부스트+하드 필터 패턴 재사용 | ~$1, 벡터 안 건드림 | 200건 샘플로 먼저 검증 가능, 실패해도 컬럼 하나 버리면 끝 |
| B | 원문 대신 "유형+핵심과업" 위주 짧은 요약을 만들어 재임베딩 | ~$2.35, 6,032건 전량 | 도메인 반복 희석 + 쿼리·코퍼스 길이 비대칭도 해소되지만 결과는 다 만들어야 확인 가능 |
| C | 유형 전용 벡터 컬럼을 따로 만들어 이중 벡터로 가중 합산 | 스키마·인덱스·쿼리 2배 | 이 규모(6천 건)엔 과함 |
| D | L1 관련도 점수(위 완료분)와 벡터 점수를 RRF로 합산 | — | 입력이 공고문 통짜라 실질적으로 A의 유형분류가 선행돼야 함 |

**재개 조건:** A부터 — 유형 분류 체계(축·개수) 확정 → 200건 샘플 분류 → 여행/유지보수 쿼리로 전후 비교 → 통과하면 전량(6,032건) 실행.

### 임베딩 인덱스는 두 개 — 뭉치면 둘 다 안 됨
| | **A — 프로젝트 (유사사례/L2)** | **B — 청크 (RAG/L3)** |
|---|---|---|
| 단위 | 프로젝트 1건 = 벡터 1개 | 텍스트 조각 = 벡터 1개 |
| 텍스트 | 제목+공고문 | Q&A 요약·매니저 노트·녹취 요약 |
| 저장 | `projects.embedding` | **새 청크 테이블 필요** |
| 상태 | ✅ **완료 (2026-07-17)** | 대기 — L3 어시스턴트 착수 시 |

"이 PG 썼을 때 뭐가 문제였나"의 답은 프로젝트 레코드가 아니라 **댓글 본문 안에** 있다(→B). "비슷한 프로젝트"는 청크로 못 찾는다(→A). B의 재료: qna 요약(`ai_insights.qna_summary`)의 clean 텍스트가 원본 노이즈 댓글보다 나은 임베딩 소스.

### 특약(subcontracts) — 계약 숫자만 (2026-07-15 확정)
지금 계약 정보는 `contractAmount` 칩(총액) 하나뿐. 가져올 것(전부 숫자·날짜, AI/파서 불필요):
- `sub_contract_subcontract.total_price`/`date_contracted` — 특약별 금액·체결일
- `milestone_milestone.contract_date_*` vs `date_*` — 차수별 예정 vs 실제(지연 분석)

**같이 고칠 버그 — `agreement_price` 서브쿼리 조건 불일치.** [`projects_incremental.sql`](./n8n/projects_incremental.sql)에서 금액은 `hide=0, date_deleted IS NULL` + `ORDER BY a.id DESC LIMIT 1`로 뽑는데, `has_valid_agreement`는 유효 특약(`is_incomplete_addon=0 AND is_cancel_addon=0`)까지 확인 → 조건 달라 agreement 여러 개면 **0원 껍데기를 고를 수 있음.** 금액을 유효 특약 `SUM(sc.total_price)`로 유도하면 증액 자동 반영 + 과업 팽창률 덤. 집계 시 0/NULL 분모 제외, **평균 아닌 중앙값+사분위수**(계약금액 롱테일).

### 선정 파트너 (싸다 — 새 워크플로 불필요, 사내망 필요)
`agreement_agreement.partner_id` → `partners_partners`의 `grade`(prime/pro/boost)·`rating`·`team_size`·`project_accepted`·`job_slug`. `projects_incremental.sql`에 몇 줄 추가. **왜:** "왜 깨졌나"의 절반은 파트너 쪽 — "팀 1명 파트너 붙은 고액 건 완료율", "신규 vs prime 취소율"이 SQL만으로 나옴. 이게 되면 타임라인 '진행 착수=선정 마커' 한계(진짜 계약체결일·파트너)가 풀림.
**결과물(산출물)은 안 가져옴** — 본진에 산출물 테이블 자체가 없음.

### 매니저 코멘트 — 비용은 병목 아님 (2026-07-14 계산)
노트 129,000건 / 프로젝트 5,993 = 프로젝트당 21건. **노트당 1콜 아니라 프로젝트당 1콜**(21개 묶어서) → 5,993콜. Haiku 4.5 Batch로 1회성 전량 ~$25. 진짜 병목은 품질·스키마.
**설계 원칙 — 요약 아니라 추출.** `{이슈유형, 발생단계, 원인태그, 심각도, 근거문장}` 고정 스키마여야 SQL 집계됨(자유 텍스트 요약은 검색·집계 안 됨). ※ qna 요약이 이 원칙을 이미 적용한 첫 사례.
**AI 전에 0원 필터:** `SELECT note_type, flag, count(*) FROM management_managenote GROUP BY 1,2` → 정산·행정(`flag`=bill/deposit/remittance), 자동생성(`note_type`=history/checklist) 제외 후 본문 앞 30자 GROUP BY로 빈발 정형문 제거.

> **관통 원칙:** 날짜·금액·상태·퍼널·계약률은 SQL이 더 정확하고 공짜 — AI 안 씀. AI는 SQL이 못 대는 자연어(노트·Q&A·통화요약·공고문)에만, 그것도 고정 스키마 추출로.
